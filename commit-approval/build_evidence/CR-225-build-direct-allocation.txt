CR-224 kmutil regression
------------------------
CR-224 introduced a same-kext class
`class AirportItlwmIO80211Packet : public IO80211NetworkPacket`. Despite
nm reporting "all 922 undefined symbols resolve against BootKC", kmutil
rejected the kext at AuxKC build time with:

    Could not use 'com.zxystd.AirportItlwm' because: Malformed vtable.
    Super class '__ZTV20IO80211NetworkPacket' has 72 entries vs subclass
    '__ZTV25AirportItlwmIO80211Packet' with 69 entries

Root cause: the local IO80211NetworkPacket / IOSkywalkNetworkPacket /
IOSkywalkPacket headers do not declare every virtual the actual Tahoe
classes have. Slot accounting at the kmutil layer is stricter than
the link-time symbol resolution our build verifier performs. The local
IOSkywalkNetworkPacket header declares only 1 IOSkywalkNetworkPacket
virtual (`getPacketType const`), while the Tahoe-26.x class actually has
`setWakeFlag`, `getTraceID const`, `setTraceID`, `traceEvent`, etc. at
intermediate vtable slots. Aligning headers slot-for-slot with Tahoe
would require reverse-engineering and patching every IOSkywalk virtual
class our kext touches — well outside the scope of any single CR.

CR-225 alternative path
-----------------------
Instead of subclassing IO80211NetworkPacket and constructing a new
vtable from incomplete headers, CR-225 allocates an actual
IO80211NetworkPacket directly using the framework's exported
allocation primitives:

  1. `IO80211NetworkPacket::operator new(size_t)` thunk at
     0xffffff80022c591c — capstone disasm:
        push rbp; mov rbp, rsp
        mov rsi, rdi                ; size_t
        lea rdi, [rip + 0x13f116]   ; IO80211NetworkPacket kalloc_type_view
        pop rbp
        jmp 0xffffff8000a4ca30      ; tail call to kalloc_type_impl
     This bypasses MetaClass::alloc() (which Apple hardcoded to NULL —
     decompiled at 0xffffff80022c5914 as `xor eax, eax; ret`) and
     allocates from IO80211NetworkPacket's own kalloc_type_view.

  2. C1 constructor `IO80211NetworkPacket(OSMetaClass const *)` at
     0xffffff80022c5840 — capstone disasm:
        push rbp; mov rbp, rsp; push rbx; push rax
        mov rbx, rdi
        call 0xffffff800297ee6a    ; IOSkywalkNetworkPacket(OSMetaClass*)
        lea rax, [rip + 0x11d0f3]  ; IO80211NetworkPacket vtable address
        mov [rbx], rax             ; install vtable
        ; restore + ret
     Chains parent ctors and installs the canonical IO80211NetworkPacket
     vtable. Constructor is exported (T global) and reachable from this
     kext.

  3. `IO80211NetworkPacket::metaClass` static (gMetaClass equivalent)
     at 0xffffff80023f86e0 — exported S global.

  4. After construction, dispatch the inherited vt[35] init virtual
     and on failure the vt[5] virtual destruct, mirroring
     `IOSkywalkNetworkPacket::withPool` body 1:1.

No same-kext subclass. No vtable construction in our kext. No header
alignment surgery. The produced object IS a real IO80211NetworkPacket —
not a sibling, not a subclass — with the canonical Tahoe-26.x
IO80211NetworkPacket vtable (so all 72 vtable entries match the framework
expectation, no kmutil mismatch possible).
