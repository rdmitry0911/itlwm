CR-231 reviewer-flagged blocker fix
------------------------------------
CR-231 was rejected because the carried CR-230 `cr230LogPayloadHex()`
unconditionally evaluated `p[row+0]..p[row+15]` for the final row at
`row=0xa0`, reading 12 bytes (`p[0xa4..0xaf]`) past the 0xa4-byte
`apple80211_wcl_connect_complete_event` payload object. Each of those
reads was a kernel-context out-of-bounds read.

CR-232 replaces the dumper with a safe variant `cr232LogPayloadHex()`:

```c
static void
cr232LogPayloadHex(const apple80211_wcl_connect_complete_event *payload)
{
    const uint8_t *p = reinterpret_cast<const uint8_t *>(payload);
    const size_t plen = sizeof(*payload);  // 0xa4
    for (size_t row = 0; row < plen; row += 16) {
        #define BYTE_AT(off) (((row + (off)) < plen) ? p[row + (off)] : (uint8_t)0)
        XYLog("DEBUG CR230_PAYLOAD off=0x%02zx ...", row,
              BYTE_AT(0), ..., BYTE_AT(15));
        #undef BYTE_AT
    }
}
```

`BYTE_AT(off)` returns 0 if `row + off >= sizeof(*payload)`. Every
dereference is bounded; the format-arg layout is preserved so the log
line is always exactly 16 hex pairs. The final row at `row=0xa0`
prints `p[0xa0..0xa3]` real bytes followed by 12 zero pads.

The unsafe `cr230LogPayloadHex()` symbol is no longer present in the
binary (replaced, not coexisting). nm output:

```

0000000000066ac0 t (anonymous namespace)::cr232LogPayloadHex(apple80211_wcl_connect_complete_event const*)
```

The `CR230_PAYLOAD` log tag is preserved (consumer-facing format
unchanged) so Stage 2 evidence aligns with prior expectations.
