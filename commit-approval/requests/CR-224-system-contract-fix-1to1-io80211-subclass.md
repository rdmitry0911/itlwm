## CHANGES_VS_CR-223

CR-224 is the smallest possible CR-223 fix per the CR-223 reviewer's
required-changes bullets:

1. **Parent class**: changed from `IOSkywalkNetworkPacket` (sibling of
   IO80211NetworkPacket) to `IO80211NetworkPacket` directly. The produced
   object is now a true `IO80211NetworkPacket` subclass; `dynamic_cast`
   /`OSDynamicCast(IO80211NetworkPacket, p)` works; `reinterpret_cast`
   in local code becomes a valid C++ upcast.

2. **Zero virtual overrides**: removed CR-223's `getPacketType()` override.
   The full IO80211NetworkPacket virtual surface (16 virtuals enumerated
   in `docs/reference/AppleBCMWLAN_io80211_network_packet_virtual_surface_2026_04_27.md`
   plus the inherited IOSkywalkNetworkPacket / IOSkywalkPacket surface)
   is inherited 1:1 by virtue of single-inheritance from IO80211NetworkPacket
   with no overrides. Vtable layout is identical to IO80211NetworkPacket's
   vtable.

3. **Constructor chain**: by deriving from IO80211NetworkPacket directly,
   the C++ constructor chain now runs `IOSkywalkPacket -> IOSkywalkNetworkPacket
   -> IO80211NetworkPacket -> AirportItlwmIO80211Packet`, matching the
   AppleBCMWLAN reference (`AppleBCMWLANPCIeSkywalkPacket` constructors call
   the IO80211NetworkPacket constructor before installing the Apple packet
   vtable).

4. **Local vtable-slot proof** (CR-223 reviewer's required change bullet 2):
   provided in `analysis/cr224_vtable_proof_2026_04_29.txt`. Source: parsed
   external relocation table for `__ZTV25AirportItlwmIO80211Packet` from
   the reviewed binary. All required slots are at their expected offsets
   pointing to the BootKC IO80211NetworkPacket / IOSkywalkNetworkPacket /
   IOSkywalkPacket exported implementations.

5. **Header revert**: `include/Airport/IO80211NetworkPacket.h::getPacketType`
   reverted from `const APPLE_KEXT_OVERRIDE` (CR-223) to non-const without
   `APPLE_KEXT_OVERRIDE`. BootKC exports the IO80211NetworkPacket-specific
   getPacketType as `__ZN20IO80211NetworkPacket13getPacketTypeEv` (NON-const,
   at vt[56]); the const-mangled `__ZNK22IOSkywalkNetworkPacket13getPacketTypeEv`
   is a SEPARATE inherited slot at vt[53]. Restoring non-const matches
   BootKC vtable layout.

## CLAIM_SCOPE

Replace `AirportItlwmIO80211PacketPool::newPacketWithDescriptor`'s body
to allocate a same-kext class `AirportItlwmIO80211Packet` derived from
`IO80211NetworkPacket`, which:

1. **Is-a `IO80211NetworkPacket`** (single inheritance, zero overrides).
   Every IO80211NetworkPacket virtual is dispatched to the IO80211Family-
   exported implementation by inherited vtable slot. The framework's
   `IO80211InfraInterface::inputPacket`, `IO80211PeerManager::inputPacket`,
   `IO80211PeerManager::skywalkInputPacket` see an object whose vtable
   is byte-equivalent (modulo gMetaClass/typeinfo header) to a vanilla
   IO80211NetworkPacket.

2. **Bypasses the alloc gate** by routing `OSTypeAlloc(AirportItlwmIO80211Packet)`
   through OUR same-kext `MetaClass::alloc()` (auto-generated by
   `OSDefineMetaClassAndStructors`) which uses our kext's
   `kalloc_type_view`. This is the only structural difference from a
   vanilla IO80211NetworkPacket — alloc path only, not vtable identity.

3. **Allocates and binds** via a `withPool` factory that mirrors
   `IOSkywalkNetworkPacket::withPool` (BootKC `0xffffff800297effa`):
   `OSTypeAlloc(AirportItlwmIO80211Packet)` (our kalloc-bypassing
   metaclass) + virtual init at `vt[35]` + virtual destruct at `vt[5]`
   on init failure. Both slots are inherited 1:1 from IOSkywalkPacket
   via IOSkywalkNetworkPacket via IO80211NetworkPacket; the dispatched
   implementations are the same as the framework factory uses.

## CONSTRUCTOR_CHAIN

`OSDefineMetaClassAndStructors(AirportItlwmIO80211Packet, IO80211NetworkPacket)`
generates a constructor that calls `IO80211NetworkPacket(OSMetaClass*)`,
which calls `IOSkywalkNetworkPacket(OSMetaClass*)`, which calls
`IOSkywalkPacket(OSMetaClass*)`, which calls `OSObject(OSMetaClass*)`.

The kext's nm output proves this:
```
T AirportItlwmIO80211Packet::AirportItlwmIO80211Packet()
T AirportItlwmIO80211Packet::AirportItlwmIO80211Packet(OSMetaClass const*)
U IO80211NetworkPacket::IO80211NetworkPacket(OSMetaClass const*)
   <- ctor-chain reaches IO80211NetworkPacket exported ctor
U IOSkywalkNetworkPacket::IOSkywalkNetworkPacket(OSMetaClass const*)
   <- ctor-chain reaches IOSkywalkNetworkPacket exported ctor (transitively)
U IOSkywalkPacket::IOSkywalkPacket(OSMetaClass const*)
   <- exported ctor
```

This matches the reference docs' assertion that AppleBCMWLAN's leaf packet
calls IO80211NetworkPacket constructor before installing its own vtable;
our subclass's construction does exactly that and installs OUR vtable
(which equals IO80211NetworkPacket's vtable layout 1:1, see vtable proof
above).

## SCRATCH_LAYER_RECONCILIATION

CR-223 reviewer's scratch-layer concern (Apple packet subclass at
`packet+0x78` consumed by IO80211/Apple paths):

- The `static_assert(sizeof(IO80211NetworkPacket) == 0x78, ...)` in
  `include/Airport/IO80211NetworkPacket.h` confirms the IO80211NetworkPacket
  layout ends at byte 0x78. Apple's packet subclass extends past 0x78
  with a per-Apple scratch buffer. Our subclass adds NO additional
  fields; `sizeof(AirportItlwmIO80211Packet) == 0x78`.

- For the **IO80211 input path** (the only path our kext drives), the
  primary STA forwarder explicitly passes a `packet_info_tag *` separately
  (CR-178/179): `inputPacket(IO80211NetworkPacket *packet,
  packet_info_tag *tag, ...)`. Tag is NOT read from packet scratch.
  Reference doc
  `docs/reference/AppleBCMWLAN_io80211_network_packet_virtual_surface_2026_04_27.md`
  Section 3 confirms "explicit `packet_info_tag *` handoff makes every
  scratch field irrelevant for the current primary-STA IO80211 input path".

- The **TX completion path** consumes `firmwareToHostTxStatus` /
  `setFirmwareTxStatus` / `getFirmwareTxStatus` virtuals only — those
  are inherited from IO80211NetworkPacket (vt[64..66] in our vtable
  proof) and dispatch to IO80211Family-exported implementations that
  use IO80211NetworkPacket-internal storage at offsets `< 0x78`.

- No Apple-PCIe-leaf scratch consumer is present in the local code or
  the local-driven framework path. Our class is intentionally a sibling
  of `AppleBCMWLANPCIeSkywalkPacket`, not a substitute for it.
