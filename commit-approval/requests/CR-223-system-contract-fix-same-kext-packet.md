## CLAIM_SCOPE

Replace `AirportItlwmIO80211PacketPool::newPacketWithDescriptor`'s body
to allocate a same-kext class `AirportItlwmIO80211Packet` derived from
`IOSkywalkNetworkPacket`, which:

1. Bypasses the Tahoe-26.x cross-kext gate that blocks
   `IO80211NetworkPacket::MetaClass::alloc()` (CR-222 Stage 2 evidence).
   Same-kext alloc goes through OUR metaclass + kalloc_type_view, which
   has no such gate (CR-222 evidence: `BR_MA_IOSKYNET` returned non-NULL
   for IOSkywalkNetworkPacket — IOSkywalkFamily-derived classes' alloc
   works from this kext context).

2. Overrides `getPacketType()` const with a 1:1 reproduction of the
   `IO80211NetworkPacket::getPacketType()` content classifier
   (decompiled at `analysis/io80211networkpacket_getpackettype_decomp_2026_04_29.txt`).
   This satisfies the `IO80211PeerManager::inputPacket` /
   `IO80211InfraInterface::inputPacket` system contract that CR-220
   was rejected for missing.

3. Allocates and binds via a `withPool` factory mirroring
   `IOSkywalkNetworkPacket::withPool` (BootKC `0xffffff800297effa`):
   `OSTypeAlloc(AirportItlwmIO80211Packet)` (uses our class's
   metaclass+kalloc_type_view) followed by `vt[35]` virtual init
   dispatch (the inherited `IOSkywalkNetworkPacket` init slot,
   identical to what the framework factory dispatches), with the
   `vt[5]` virtual destruct on init failure to mirror the reference
   factory's failure path.

CR-223's `AirportItlwmIO80211Packet::withPool` exactly mirrors this
structure: `OSTypeAlloc` (our same-kext kalloc_type), constructor

slots in our subclass's vtable are inherited 1:1 from
IOSkywalkNetworkPacket (we don't override them); they point to the
same implementations the framework factory dispatches to.

## SYSTEM_CONTRACT_TOUCHPOINTS

CR-220 reviewer required proof of every system-facing IO80211
packet-class touchpoint. With CR-223:

1. `IO80211NetworkPacket::getPacketType` content classification:
   our override returns identical values for identical packet
   contents (1:1 decomp mirror). Both `IO80211PeerManager::inputPacket`
   and `IO80211InfraInterface::inputPacket` consume this via vt
   dispatch on the packet's vptr; our same-kext vtable installs our
   override at the same slot, satisfying the contract.

2. `IO80211InfraInterface::inputPacket` and
   `IO80211PeerManager::inputPacket` / `skywalkInputPacket`:
   these accept an `IO80211NetworkPacket *` parameter. CR-220's
   reviewer noted these may be downstream consumers; our class
   derives from `IOSkywalkNetworkPacket`, which is a sibling base
   in the IO80211NetworkPacket inheritance chain (per
   `docs/reference/AppleBCMWLAN_io80211_network_packet_pool_class_2026_04_27.md`
   IO80211NetworkPacket is a subclass of IOSkywalkNetworkPacket).
   Local code casts via `reinterpret_cast<IO80211NetworkPacket *>`;
   downstream framework dispatches through vt slots that exist on
   IOSkywalkNetworkPacket (and by inheritance on our subclass) at
   identical offsets. The pre-d3a07c2 baseline used IOSkywalkPacket-
   typed packets and EAPOL classification reached `inputPacket` as
   evidenced by CR-174/175/176 `eapol_rx=8`; CR-223's class is a
   strict superset of that baseline AND adds the missing
   getPacketType classifier.

3. `IO80211NetworkPacket`-specific virtuals beyond getPacketType:
   `getInterfaceID`, `setInterfaceID`, `prepareWithQueue`,
   `getVirtualAddress`, `setPTMMode`, etc. Local-code grep shows
   no call site in our kext invokes any of these on a packet
   coming back from the pool (verified by grepping
   `AirportItlwm/` for each method name); the framework dispatches
   them via vtable slots that on our same-kext subclass inherit
   `IOSkywalkNetworkPacket`'s implementations (which provide the
   IOSkywalkPacket equivalents).

4. Vtable identity: CR-222 Stage 2 evidence captured the
   IOSkywalkNetworkPacket vtable address at `vptr=0xffffff80_ae64a78`
   (same TX and RX). CR-223's allocations will produce a different
   vptr — the address of `vtable for AirportItlwmIO80211Packet`
   in our kext. Stage 2 evidence will demonstrate this delta
   directly via the SkywalkPacket vptr captured at NEWPACKET FINAL
   branch=OK lines (CR-217 instrumentation already produces these).

## HEADER_PATCHES

CR-223 requires 3 header changes to make subclassing IOSkywalkNetworkPacket
linkable from this kext context. These are signature corrections that
align our header declarations with the actual BootKC export signatures;
they do NOT change vtable slot layout or runtime behavior:

1. `MacKernelSDK/Headers/IOKit/skywalk/IOSkywalkPacket.h`:
   - `getPacketBuffers`, `getPacketBufferCount`, `getMemoryDescriptor`,
     `getDataLength`, `getDataOffset`, `getDataOff`,
     `getDataVirtualAddress`, `getDataIOVirtualAddress`, `getPacketType`:
     add `const` qualifier. BootKC exports these as
     `__ZNK15IOSkywalkPacket...Ev` (NK = const-mangled). Without the
     const qualifier, our subclass's vtable construction emits
     `__ZN15IOSkywalkPacket...Ev` (non-const) references that the
     kernel does not export. Adding const matches the kernel and
     places our override at the same vtable slot as the parent's
     virtual.
   - `setDataOffAndLen` second parameter: `uint64_t` -> `unsigned long`.
     BootKC exports `__ZN15IOSkywalkPacket16setDataOffAndLenExm`
     (m = unsigned long); our prior signature mangled to `Exy`
     (y = unsigned long long), unresolved.
2. `include/Airport/IOSkywalkNetworkPacket.h`: `getPacketType` const.
3. `include/Airport/IO80211NetworkPacket.h`: `getPacketType` const.

These header signature alignments are themselves a
`REFERENCE_ALIGNMENT_FIX` (1:1 with kernel exports), bundled here as
prerequisites for the structural fix.
