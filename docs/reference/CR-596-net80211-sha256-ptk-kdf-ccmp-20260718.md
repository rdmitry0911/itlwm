# CR-596 — net80211 SHA-256 PTK KDF CCMP correction

Date: 2026-07-18

## Scope

This correction is limited to the SHA-256 KDF branch selected by
ieee80211_is_sha256_akm() when deriving a CCMP PTK. It corrects the KDF label
length and the requested derived-key length. The legacy SHA-1 PRF branch
retains its label separator and 64-byte local PTK carrier.

## Derivation contract

The SHA-256 KDF input is:

~~~text
LE16(counter) || "Pairwise key expansion" || Context || LE16(L)
~~~

For CCMP, L is 384 bits and the output is the first 48 bytes:
KCK[16] || KEK[16] || TK[16]. Unlike the SHA-1 PRF, the SHA-256 KDF does not
consume a NUL label separator before Context; it carries the requested bit
length explicitly.

## Local correction

ieee80211_derive_ptk() now derives:

- SHA-256 KDF: sizeof(ptk_label) - 1 and MIN(48, sizeof(*ptk));
- legacy SHA-1 PRF: sizeof(ptk_label) and sizeof(*ptk).

The ordered AA/SPA and ANonce/SNonce context construction, the KDF selector,
and the local PTK storage layout are otherwise unchanged.

## Boundaries

This is a PSK-SHA256/CCMP PTK derivation correction. It does not claim coverage
for SAE, TKIP, group-key derivation, management-key derivation, firmware
behavior, CoreCapture fault reporting, or Apple private Wi-Fi framework result
codes.

## Deterministic guard

scripts/test_net80211_sha256_ptk_kdf_contract.sh computes a synthetic
no-secret HMAC-SHA256 vector using only Python's standard library. It checks
the 48-byte result, AA/SPA and nonce ordering invariance, the retained
64-byte SHA-1 PRF vector, and the decisive local source anchors.
